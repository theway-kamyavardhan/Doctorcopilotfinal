DOCTORCOPILOT - PROJECT INTRODUCTION

Purpose
- DoctorCopilot is an AI-assisted healthcare platform for patients, doctors, and administrators.
- It converts uploaded medical reports into structured health data, trends, insights, anomalies, and case workflows.
- The system is built to help patients understand their health better and help doctors review patients faster with historical context.

Core Goal
- Upload reports once.
- Process them once.
- Store the extracted medical information in the database.
- Reuse that stored information everywhere instead of recomputing it repeatedly.


1. MAIN USER ROLES

Patient
- Registers and logs in with a Patient ID such as P-10001.
- Uploads reports.
- Views dashboard, reports, timeline, trends, cases, chats, calendar, and settings.
- Can request consultation with a selected doctor.
- Can approve or deny report access requests from doctors.

Doctor
- Logs in with a Doctor ID such as D-10001.
- Reviews pending consultation requests.
- Accepts, rejects, refers, or opens cases.
- Chats with patients.
- Requests access to patient reports.
- Reviews patient trends, insights, anomalies, linked reports, and schedules appointments.

Admin
- Logs in with ADMIN-001.
- Manages doctors, patients, system state, cases, reports, and operational overview.


2. HIGH-LEVEL ARCHITECTURE

Frontend
- React + Vite
- Tailwind CSS
- Framer Motion
- SWR for caching selected patient data
- Route-based portals for patient, doctor, and admin

Backend
- FastAPI
- Async SQLAlchemy
- JWT-based authentication
- Role-based route protection
- WebSocket chat for live case communication

Database
- Supabase PostgreSQL
- Stores users, patients, doctors, reports, extracted report data, cases, messages, notes, and appointments

File Storage
- Supabase Storage bucket: reports
- Stores original uploaded medical report files

AI / Processing
- OCR and text extraction happens during report upload
- Parameter extraction and insight generation happen once during processing
- Later reads reuse stored values instead of calling AI again


3. STRICT ID SYSTEM

Patient IDs
- Format: P-10001, P-10002, ...

Doctor IDs
- Format: D-10001, D-10002, ...

Admin ID
- Format: ADMIN-001

Important
- Existing IDs are preserved.
- The system does not regenerate old IDs after migration.
- Login uses these public IDs directly.

5. COMPLETE PROJECT FLOW

Step 1: Authentication
- A user logs in through FastAPI auth.
- Backend validates ID and password.
- Backend returns JWT containing user_id and role.
- Frontend stores token and role and routes the user to the correct portal.

Step 2: Report Upload
- Patient uploads a report file.
- Backend stores the original file in Supabase Storage.
- Backend extracts text using direct PDF parsing or OCR if needed.
- Backend parses medical parameters from the report.
- Backend stores extracted values in the database.
- Backend generates and stores insights for that report.

Step 3: Data Reuse
- Patient dashboard, trends, timeline, doctor insights, and exports read stored values from the database.
- The system avoids recomputation for normal viewing.

Step 4: Consultation Request
- Patient selects a doctor and sends a consultation request.
- A duplicate active consultation with the same doctor is not allowed until the current one is closed.
- Request appears in the doctor workflow.

Step 5: Doctor Review
- Doctor sees pending cases.
- Doctor can:
  - accept
  - reject
  - refer to another doctor
  - view overview
- Once accepted, the case becomes active.

Step 6: Report Access
- Doctor requests patient report access through case/chat flow.
- Patient approves or denies.
- If approved, doctor can open linked reports inside the case.

Step 7: Doctor Review of Reports
- Doctor can view:
  - AI summary
  - original uploaded report
  - PDF export options
- Doctor can also open a dedicated insights route for trends, anomalies, and patient-level review.

Step 8: Case Collaboration and Closure
- Doctor can consult, chat, and manage the case.
- Closed cases move into archived workflow sections.
- Archived cases can be deleted by doctor-side archive actions where allowed.

Step 9: Appointment Flow
- Doctor schedules an appointment with an accepted patient case.
- Appointment appears on doctor dashboard/calendar.
- Appointment appears on patient dashboard/calendar.

10. AI / MEDICAL INTELLIGENCE LOGIC

Important Principle
- AI processing happens mainly at upload time.
- Later dashboards and views reuse stored results.

What gets generated from uploaded reports
- extracted values
- report summary
- anomalies
- trend-relevant structured data

What trend system does
- compares historical parameter values across reports
- identifies up/down/stable patterns
- supports patient and doctor insights pages

What anomaly system does
- detects persistent low/high patterns
- detects sudden change
- highlights clinically important values

What is not done repeatedly
- the system does not re-run full AI generation for every page load
- doctor insights route is read-only and cost-aware

14. WHAT TO SAY IN A VIVA

Short Introduction
- DoctorCopilot is an AI-assisted healthcare platform that lets patients upload medical reports, extracts structured health data, stores that data in Supabase, and then reuses it across dashboards, trends, doctor case review, chat, and appointment workflows.

Backend Explanation
- The backend is built with FastAPI, uses JWT authentication, connects to Supabase Postgres for persistent data, and Supabase Storage for medical report files.

Frontend Explanation
- The frontend is built using React and Vite, with role-based portals for patient, doctor, and admin users.

AI Explanation
- AI and OCR happen mainly when a report is uploaded. The extracted values and summaries are saved so later views do not need costly recomputation.

Clinical Workflow Explanation
- A patient uploads reports, requests consultation, a doctor accepts the case, requests report access, reviews insights and reports, chats with the patient, and schedules appointments. Closed cases move into archive.
