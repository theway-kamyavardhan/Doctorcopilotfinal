# DoctorCopilot: Doctor Workspace & UI Architecture

This document tracks the complete clinical workspace built for DoctorCopilot, detailing the entire routing, components, features, and workflows.

---

## 1. Global Navigation & Layout 

**Component:** `DoctorLayout.jsx`
The Doctor workspace is embedded in a persistent, Apple-style glassmorphic layout (`backdrop-blur`). 
It features a static auto-expanding Left Navigation sidebar routing the doctor to:
- **Dashboard** (`/doctor/dashboard`): The primary landing node.
- **Cases** (`/doctor/cases`): The clinical triage inbox.
- **Chats** (`/doctor/chats`): Dedicated messaging matrix (Active).
- **Calendar** (`/doctor/calendar`): Sync and scheduling.
- **Archive** (`/doctor/archive`): Historical case parameters.

When viewing a specific case (`/doctor/case/:id`), the global layout cleanly hides the sidebar to dedicate 100% of the screen real-estate to the 3-panel triage layout.

---

## 2. Doctor Dashboard (`DoctorDashboardView.jsx`)

The default landing page providing a system-wide snapshot of the doctor's immediate responsibilities.
- **Dynamic Greeting & Profile**: Extracts the doctor's name, specialization, and hospital from `getDoctorProfile()`.
- **Top-Level Metrics Strip**: Renders statistics based on live database queries for `Pending Requests`, `Active Cases`, `Unread Threads`, and `Reports Linked`.
- **Live Consultation Requests**: An active feed displaying cases requiring evaluation. Doctors can click **"Accept & Open"** directly from the dashboard which initializes the case status to `open` and routes them to the Case View.
- **Schedule Widget**: A side-panel rendering upcoming synchronous clinical handovers and patient reviews mapped from `getDoctorAppointments()`.
- **Primary CTA ("New Consultation")**: Triggers the system modal to securely launch a track.

---

## 3. New Consultation Modal (`NewConsultationModal.jsx`)

An absolute-positioned glassmorphic overlay launched from the Dashboard.
- **Step 1 (Patient Lookup)**: Initiates a live database search (`searchDoctorPatients`) querying existing registered identifiers or names.
- **Step 2 (Case Creation)**: Requires the doctor to draft a Case Title and Reason.
- **Automated Handshake**: Upon clicking "Create Case," the API dynamically generates the consultation pipeline and maps the exact doctor payload to the patient, ensuring both parties are immediately synced.

---

## 4. Cases Inbox (`CaseInbox.jsx`)

A highly sophisticated, tabbed triage board.
- **Dynamic Tabs**: `active`, `pending`, and `archived`. Automatically maps FastAPI case status data (e.g., `in_review`, `transferred`, `closed`).
- **Data Rendering**: Each card loops through:
  - Patient Name & ID.
  - Generative AI Summary (evaluates recent anomalous metrics).
  - Time since last update.
  - "Accept" action buttons for pending cases or "Open" routing arrows for active scenarios.
- **Live Refreshing**: The inbox polls the backend securely every 12 seconds (`setInterval`) silently re-validating the queue without dropping the doctor's UI state.

---

## 5. The Ultimate 3-Panel Case View (`DoctorCaseView.jsx`)

The crown jewel of the platform housing the core clinical operations when evaluating `/doctor/case/:id`.

**PANEL 1: Clinical Chronology (Left Panel - 25% width)**
- Renders a clean vertical timeline parsing every event linked to the case.
- Categorizes events into `ai_flag` (Alerts) or `report_logged` (Standard Uploads).
- Tracks Patient demographics (Age/Gender/Blood Group).

**PANEL 2: AI Clinical Engine (Center Panel - Auto)**
- **Health Score Orbit**: Uses a horizontal progress track mapping a `/100` health score defined by the severity of the flagged laboratory parameters.
- **Trend Confidence**: AI asserts its own confidence (e.g., `92%`) depending on how many historical reports it can triangulate.
- **"Explain Case Parameters" (Generative Call)**: Features an Apple Interface `framer-motion` sequence that cascades the AI’s summary into three distinct visual blocks:
    - **Generative Diagnosis** (The clinical evaluation)
    - **Identified Risks** (e.g., Hepatic Failure flags, low platelet counts)
    - **Clinical Recommendations** (suggested medical actions mapped to the risk tags).

**PANEL 3: Secure Terminal & Actions (Right Panel - 30% Width)**
- **Action Strip**: Instant commands mapping specific `doctor.service.js` patch calls.
    - **Transfer**: Securely moves the case block off the current doctor ledger.
    - **Invite**: Triggers multi-doctor network collaboration.
    - **Close Case (FileLock)**: Requires generating a final system note acknowledging the case resolution before the backend seals the interaction.
- **Embedded WebSockets Terminal**: A chat window directly interfacing with `app.websocket("/ws/cases/{case_id}")`. It handles live messaging between the Patient App and the Doctor UI simultaneously, maintaining clinical context alongside the diagnostic parameters.

---

## 6. Service & API Layer (`doctor.service.js`)

A heavily encapsulated API binder providing cleanly formatted normalization across the system.
If the API returns raw, decoupled variables, the `normalizeCase` pipeline maps it into:
- Defined Prioritization logic (`pending` = critical, `open` = moderate).
- Abnormal parameter clipping (`buildWatchlistFromReports`).
- Automated Generative prompt handling (`buildAiExplanation`).
This ensures the UI components remain completely agnostic of backend structural schemas, purely subscribing to clean JSON objects like `case.patientName` or `case.health_score`.

---

### UI/UX Design System Notes
- Relies heavily on `backdrop-blur-xl` properties mapping over deep radial linear gradients.
- Typography strictly follows uppercase `track-[0.2em]` styling for eyebrows/subtitles to emphasize a tactical, "military/science" level of UI seriousness.
- Extensive micro-animations driven by `framer-motion` (lists staggering into view, buttons pulsating upon API sync).
