# DOCTORCOPILOT — ENTERPRISE SYSTEM ARCHITECTURE OVERVIEW

## 1. CORE TECHNOLOGY STACK
DoctorCopilot is engineered as a modern, high-fidelity, dual-pane (Patient/Doctor) healthcare platform. It utilizes a decoupled, API-first architecture.

### **Frontend Stack (The "Glass" Layer)**
*   **Framework**: React 18 / Vite 5
*   **Routing**: React Router DOM (v7) 
*   **Styling Engine**: Tailwind CSS 4.x (Native PostCSS)
*   **UI/UX Paradigm**: Apple-inspired "Premium Glassmorphism" (heavy use of `backdrop-blur-2xl`, SVG color matrices, and chromatic aberration overlays).
*   **Animations & Micro-interactions**: Framer Motion (page transitions, state layout morphs).
*   **3D / Visualizations**: `@react-three/fiber`, `@react-three/drei` (Living System Neural Core), and `recharts` (Clinical trend visualizations).
*   **API Client**: Axios (Iterceptors configured for JWT token rotation/injection).
*   **State Management**: React Context (`ThemeContext`, `AuthContext`) + derived local states (`useMemo`, `useEffect`).

### **Backend Stack (The "Engine")**
*   **Framework**: FastAPI (Python 3.x), utilizing Starlette for WebSockets.
*   **Server**: Uvicorn (ASGI web server).
*   **Database**: SQLite (`doctorcopilot.db`) mapped via `aiosqlite` for asynchronous disk I/O.
*   **ORM Layer**: SQLAlchemy 2.0 (Async Engine) + Pydantic v2 (Data validation/serialization).
*   **Migrations**: Alembic (with custom runtime schema altering in `app/db/schema.py` to bypass SQLite limitation on `ALTER TABLE`).
*   **Authentication**: JWT Token protocol (Auth2 Password Bearer), `passlib` (Bcrypt password hashing), `python-jose`.
*   **AI / Inference**: OpenAI SDK (configured for Gemini or compatible clinical analysis engines) + custom prompt engineering for extracting medical parameters.
*   **Document Intelligence**: `pypdf`, `pdfplumber`, `pytesseract` (OCR fallback), `pdf2image`, `pillow` (for clinical report digitization).

## 2. SYSTEM ARCHITECTURE & DATA FLOW

The platform enforces a strict boundary between public marketing, clinical patient workflows, and specialized doctor triage workflows.

### **The Routing Tree**
*   **Public Space** (`/`) -> High-performance 3D cinematic landing page.
*   **Auth Space** (`/login`, `/register`) -> Role selection and dynamic JWT issuance.
*   **Patient Portal** (`/patient/*`)
    *   `/dashboard`: Summary of active stats, neural core status, recent AI insights.
    *   `/reports`: Medical document vault (upload PDFs, trigger AI background analysis).
    *   `/cases`: Track consultation requests, active cases, and appointments.
    *   `/chat`: Real-time secure WebSocket connection to doctors.
*   **Doctor Portal** (`/doctor/*`)
    *   `/dashboard`: Triage queue, pending request alerts, next-action indicators.
    *   `/cases`: Master inbox for pending, active, and archived consultations.
    *   `/case/:id`: Deep dive into a patient's historical trends, anomalies, and reports.
    *   `/calendar`: Appointment scheduling mapped directly to accepted cases.

### **The Consultation Lifecycle (State Machine)**
The platform treats every medical event as a `Case`. A case operates under a strict finite state machine executed by the backend `cases.py` endpoints:
1.  **Creation**: A patient requests a doctor (`POST /cases`) -> `status: pending`.
2.  **Referral (Alternative Origin)**: A doctor refers a case to a specialist -> `status: pending` (Origin flagged as `doctor_referral`).
3.  **Triage**: Doctor views case overview (checking anomalies/trends). 
4.  **Acceptance**: Doctor accepts -> `status: open`. Backend auto-generates a system message (`accept_notice`) inside the case chat.
5.  **Rejection**: Doctor rejects -> `status: closed`.
6.  **Real-Time Consultation**: WebSocket opens. Doctor and Patient communicate.
7.  **Evidence Lock**: By default, reports are locked. Doctor requests access (`POST /cases/{id}/report-access/request`). Patient grants access (`PATCH /cases/{id}/report-access`).
8.  **Action**: Doctor defines appointments -> `table: appointments` linked to `case_id`.
9.  **Closure**: Doctor closes the case with a final clinical note -> `status: closed` (Archived).

## 3. DATABASE SCHEMA TOPOLOGY
The relational blueprint is designed for high-referential integrity while allowing scalable AI metadata mapping.

*   `users`: Base authentication entity (`id`, `email`, `hashed_password`, `role: patient|doctor|admin`).
*   `patients`: Extends `users`. Holds demographics (`age`, `gender`, `blood_group`, `phone_number`).
*   `doctors`: Extends `users`. Holds professional data (`specialization`, `hospital`, `location`, `license_number`).
*   `reports`: Linked to `patient_id`. Stores S3/Local file paths, AI extraction payload (JSON), generated `insights` array, and parsed parameters.
*   `cases`: The central hub. Links `patient_id` <-> `doctor_id`. Holds metadata (`status`, `description`, `referral_note`, `closing_note`, `report_access_status`).
*   `case_messages`: Message ledger (`case_id`, `sender_type`, `message_type`, `content`). Handles multi-media signals (text vs system UI cards).
*   `appointments`: Calendar tracking mapping `patient_id`, `doctor_id`, and `case_id`.

## 4. THE AI INTELLIGENCE PIPELINE

The most critical backend loop runs when a patient uploads a clinical report (`POST /api/v1/patients/me/reports`).
1.  **Ingestion**: FastAPI receives the multipart/form-data PDF.
2.  **Digitization**: `pdfplumber` attempts native text extraction. If native text is absent (scanned document), `pdf2image` + `pytesseract` executes a deep OCR sweep.
3.  **Neural Parsing**: The extracted raw text is bundled into a rigid system prompt and fired to the AI Inference Engine.
4.  **Structured JSON Generation**: The LLM is forced to return a strictly typed JSON object mapping:
    *   `report_type` (e.g., Complete Blood Count).
    *   `parameters` (e.g., `["Hemoglobin", 14.5, "g/dL", "13.5-17.5", "normal"]`).
    *   `insights` (Human-readable clinical observations).
5.  **Trend Aggregation**: When the Doctor views the case, the system dynamically sweeps *all* past reports for that patient, correlating matching parameters chronologically to identify **Anomalies** (sudden spikes, persistent critical values).

## 5. REAL-TIME COMMUNICATIONS (WebSockets)

Instead of stateless polling, DoctorCopilot uses a persistent WebSocket highway for live consultations.
*   **Endpoint:** `ws://[HOST]/api/v1/ws/cases/{case_id}?token={JWT}`
*   **Handshake**: FastAPI extracts the JWT from the query param, validates the exact UUID of the user against the database, and confirms the `case_id` belongs to them.
*   **Broadcast Grouping**: The server allocates active connections into memory arrays mapped by `case_id`.
*   **Payload Execution**: When a client sends a JSON payload, FastAPI appends server-side timestamps, runs database `INSERT` on `case_messages`, and broadcasts the formatted object directly back to both ends.
*   **UI Projection**: React renders specialized UI "Cards" (Acceptance Notices, Shield Requests) depending on the `message_type` enum streamed over the socket.

## 6. DESIGN SYSTEM LOGIC

*   **Dark Mode Paradigm**: `isDark === true` activates Obsidian hues (`bg-slate-950/90`), deep Cyan shadows, and Amber/Emerald warning tones.
*   **Light Mode Paradigm**: Crisp Titanium Whites (`bg-white/85`), soft slate borders, and vibrant primary Apple-blue accents.
*   **Glass Surface Overlays**: Central architectural wrapper `GlassSurface.jsx` intercepts background light and mathematically alters the saturation and brightness using SVG native `feColorMatrix` and `feDisplacementMap` filters before piping it into a `backdrop-blur` CSS runtime property.
